# Plan: Fix Art Style Centering in Manual Mode

## Investigation Summary

The issue affects three art styles in Manual mode: **barCode**, **scatterMatrix**, and **voronoiCells**. These styles appear offset from the expected center position (0, 0) when X/Y visual dimensions are set to their default values.

### Root Cause Analysis

The renderer applies canvas transforms in `renderUsingExplicitDimensions()` before calling the style's `render()` method:

```javascript
// From renderer.js lines 629-631
this.ctx.translate(cssWidth / 2 + translateX, cssHeight / 2 + translateY);
this.ctx.rotate(rotationRad);
this.ctx.scale(scaleFactor, scaleFactor);
```

Where `translateX = (normX - 0.5) * cssWidth` (normX is -1..1, default 0 → translateX = -0.5 * cssWidth).

This moves the coordinate origin from the top-left to the **canvas center plus X/Y offset**.

### The Inconsistency

| Style | Manual Mode Positioning | Respects X/Y Offset? |
|-------|------------------------|---------------------|
| particleField (working) | `(normX - 0.5) * drawW` | YES - uses centered coords |
| voronoiCells | `(p.x - 0.5) * width` | PARTIAL - uses data point X/Y which are fixed |
| barCode | `(w - totalBarsWidth) / 2` | NO - ignores transforms, centers on canvas |
| scatterMatrix | `(w - matrixW) / 2` | NO - ignores transforms, centers on canvas |

**barCode** and **scatterMatrix** calculate their center position using the full canvas dimensions (`w`, `h`), which positions them at the **absolute canvas center** regardless of the X/Y visual dimension offsets.

**voronoiCells** uses data point positions, but the renderer generates these points around a fixed center (0.5, 0.5) with no variation based on X/Y offsets.

### Why the Reported Offsets

When X=Y=0 (default), the canvas origin is at the center, so:
- barCode centers at canvas center → appears at visual position (0, 0) ✓
- But when users move X/Y sliders, barCode stays at canvas center ✗

The reported "centered around (-0.9, -0.8)" suggests users found positions that visually center the artwork due to the disconnect between:
1. The canvas transform (which moves the origin)
2. The style's positioning (which assumes top-left origin)

## Proposed Solutions

### Option 1: Apply X/Y Offset Directly in Problem Styles (Recommended)

Modify barCode, scatterMatrix, and voronoiCells to read the X/Y offset from the data points and apply it to their positioning calculations.

**For barCode:**
```javascript
// Read offset from first data point
var offsetX = (dataPoints[0].x - 0.5) * w * 0.8; // 0.8 spread factor from renderer
var offsetY = (dataPoints[0].y - 0.5) * h * 0.8;
var startX = (w - totalBarsWidth) / 2 + offsetX;
var startY = (h - barHeight) / 2 + offsetY;
```

**For scatterMatrix:**
```javascript
var offsetX = (dataPoints[0].x - 0.5) * w * 0.8;
var offsetY = (dataPoints[0].y - 0.5) * h * 0.8;
var startX = (w - matrixW) / 2 + offsetX;
var startY = (h - matrixH) / 2 + offsetY;
```

**Pros:**
- Minimal changes to existing code
- Follows pattern of particleField (uses data point values)
- X/Y sliders will work intuitively

**Cons:**
- Requires each style to know about the offset convention
- Slight coupling between renderer data generation and style rendering

---

### Option 2: Pass X/Y Offset via renderingConfig

Extend `renderingConfig` to include explicit `offsetX` and `offsetY` values (in pixels or normalized), making the offset explicit rather than derived from data points.

**In renderer.js:**
```javascript
renderConfig.offsetX = normX;  // -1 to 1
renderConfig.offsetY = normY;  // -1 to 1
```

**In art styles:**
```javascript
var offsetX = (renderingConfig.offsetX || 0) * width * 0.5;
var startX = (w - elementWidth) / 2 + offsetX;
```

**Pros:**
- Explicit contract between renderer and styles
- Easier to understand and document
- No dependency on data point values

**Cons:**
- Requires modifying renderer and all manual-mode styles
- Slightly more complex API

---

### Option 3: Don't Apply Canvas Transform for Position

Remove the X/Y translation from the canvas transform stack and instead pass it to styles to apply in their coordinate calculations (similar to Option 2 but removes from transform).

**In renderer.js (modified):**
```javascript
// Only apply rotation and scale, NOT position translation
this.ctx.translate(cssWidth / 2, cssHeight / 2); // Always center
this.ctx.rotate(rotationRad);
this.ctx.scale(scaleFactor, scaleFactor);
// Position offset is passed via renderConfig.offsetX/Y
```

**Pros:**
- Simpler coordinate system for styles (always centered)
- Position is explicit, not implicit in transforms

**Cons:**
- Major change to renderer architecture
- All styles would need updates
- May break existing working styles

---

### Option 4: Normalize All Styles to Use Data Point Positions

Refactor barCode, scatterMatrix, and voronoiCells to iterate through data points like particleField does, using each point's X/Y to position individual elements.

**For barCode:**
Instead of one cohesive bar code, draw bars at each data point position (like particleField draws particles).

**Pros:**
- Consistent architecture across all styles
- Each element can have independent position/size/color

**Cons:**
- Changes the artistic design (barCode becomes scattered bars, not cohesive barcode)
- Not aligned with the original design intent

---

## Recommendation

**Implement Option 1** for the following reasons:

1. **Minimal impact** - Only 3 files need modification
2. **Consistent with working styles** - particleField already uses data point positions
3. **Preserves artistic intent** - barCode remains a cohesive barcode, scatterMatrix remains a grid
4. **Clear data flow** - Renderer generates points with X/Y offsets baked in, styles consume them

## Implementation Plan

### Step 1: Fix barCode.js
- In `_drawManualBarCode()`, calculate offset from `dataPoints[0].x` and `dataPoints[0].y`
- Apply offset to `startX` and `startY`
- Test with X/Y sliders at various positions

### Step 2: Fix scatterMatrix.js
- In `_drawManualMatrix()`, calculate offset from data points
- Apply offset to `startX` and `startY`
- Test with X/Y sliders

### Step 3: Fix voronoiCells.js
- Modify point generation to offset based on `dataPoints[0]` position
- Or adjust the seed point positions to incorporate X/Y offset
- Test with X/Y sliders

### Step 4: Verification
- Test all three styles at X=0, Y=0 (should center)
- Test at X=-1, Y=-1 (should move up-left)
- Test at X=1, Y=1 (should move down-right)
- Compare behavior with particleField for consistency

## Files to Modify

1. `src/canvas/styles/barCode.js` - Add X/Y offset handling in `_drawManualBarCode()`
2. `src/canvas/styles/scatterMatrix.js` - Add X/Y offset handling in `_drawManualMatrix()`
3. `src/canvas/styles/voronoiCells.js` - Add X/Y offset handling in manual mode point generation

## Success Criteria

- All three styles render centered when X=0, Y=0
- Moving X/Y sliders shifts the artwork in the corresponding direction
- Behavior is consistent with particleField and other working styles
- No regression in Data-driven mode rendering
