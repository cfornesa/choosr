# Plan: Fix Delete Button and Visual Dimensions in studio.php

## Session ID: 1777105818805-sunny-comet

## Issues Identified

### Issue 1: Missing "Delete Artwork" Button
**Current State:** The delete button exists in the Artwork Metadata panel (line 140 of studio.php) but is hidden by default (`style="display:none;"`). It only appears after loading an existing artwork.

**User Expectation:** A "Delete Artwork" button should be visible below the Save, Load, and New Artwork buttons in the main controls area.

**Assumption to Surface:** Moving the Delete button to the main controls area makes it always visible (even for unsaved/new artworks), which differs from the current design where it only shows for existing artworks. The user may want this behavior changed to always show, or may want clarification on when it should be visible/disabled.

### Issue 2: Visual Dimensions Sliders Not Applying Correctly
**Current State:** The `renderUsingExplicitDimensions` function in renderer.js generates data points scattered around the explicit dimension values, treating them as relative offsets rather than absolute transformations.

**Root Cause:** The dimensions are being applied per-data-point rather than as canvas-level transformations. For example:
- X/Y Position set the center point of the generated grid, not the entire artwork's position
- Size sets individual data point sizes, not overall scale
- Opacity is passed to data points but not applied globally
- Rotation is passed to data points but many styles don't implement it

**Expected Behavior (Per User):**
- **X Position**: Changes horizontal position of the centroid (entire artwork shifts left/right)
- **Y Position**: Changes vertical position of the centroid (entire artwork shifts up/down)  
- **Size**: Scales the entire artwork proportionally
- **Opacity**: Changes overall opacity of all elements
- **Rotation**: Rotates the entire artwork around its centroid

**Assumption to Surface:** Applying canvas-level transformations will affect ALL art styles uniformly. This may change the visual output for users who are used to the current behavior. The user should confirm they want global canvas transformations applied before rendering.

## Proposed Solution

### Approach A: Add Delete Button to Main Controls + Canvas-Level Transformations (Recommended)

This approach addresses both issues comprehensively:

1. **Delete Button**: Add a visible "Delete Artwork" button below the existing button row in studio.php. Wire it in app.js to call the delete API.

2. **Visual Dimensions**: Modify `renderUsingExplicitDimensions` in renderer.js to:
   - Save the canvas context state
   - Apply canvas transformations: translate(x, y), scale(sizeFactor), rotate(rotation), globalAlpha(opacity)
   - Render the artwork through the style module
   - Restore the canvas context

**Pros:**
- Clean separation of concerns - transformations happen in one place
- Works uniformly across ALL art styles without modifying them
- Minimal code changes
- Consistent behavior regardless of art style

**Cons:**
- Changes the existing visual behavior (which may be what the user wants fixed)
- Delete button will be visible even for new/unsaved artworks (may need disable logic)

### Approach B: Reframe - Fix at Art Style Level Instead

Instead of canvas-level transformations, modify individual art styles to properly interpret the explicit dimensions from data points.

**Pros:**
- More granular control per style
- Could preserve some existing behaviors

**Cons:**
- Requires modifying every art style file
- Inconsistent behavior likely across styles
- Much more code to maintain
- Some styles may not have rotation/shape features

### Approach C: Unexpected - Use CSS Transforms on Canvas Element

Instead of canvas context transformations, apply CSS transforms to the canvas element itself.

**Pros:**
- Very simple implementation
- GPU accelerated

**Cons:**
- Would transform the entire canvas including background
- Export PNG would not reflect the transforms (canvas.toBlob captures pre-CSS state)
- Breaks the expected export behavior

### Phase 2: Fix Visual Dimensions - Canvas-Level Transformations

**File: src/canvas/renderer.js - Function: renderUsingExplicitDimensions**

Modify the function to apply canvas transformations:

```javascript
Renderer.prototype.renderUsingExplicitDimensions = function(explicitDimensions, paletteConfig, renderingConfig, styleKey) {
  // ... existing validation code ...
  
  var cssWidth = this.canvas.width / this._dpr;
  var cssHeight = this.canvas.height / this._dpr;
  
  // Clear canvas
  this.clear(paletteConfig ? paletteConfig.background : DEFAULT_BACKGROUND);
  
  // Get style module
  var styleModule;
  try {
    styleModule = window.DataToArt.ArtStyles.getStyle(styleKey);
  } catch (e) {
    // ... error handling ...
    return;
  }
  
  // Calculate centroid position from explicit dimensions
  // X, Y are -1..1, map to canvas coordinates with padding
  var centerX = cssWidth / 2;
  var centerY = cssHeight / 2;
  var translateX = (explicitDimensions.x || 0) * (cssWidth / 2);  // -1..1 → -width/2..width/2
  var translateY = (explicitDimensions.y || 0) * (cssHeight / 2); // -1..1 → -height/2..height/2
  
  // Calculate scale from size (0..500 pixels → scale factor)
  // Use a base scale where 100px = 1.0 (identity scale)
  var baseSize = 100;
  var scaleFactor = (explicitDimensions.size || 100) / baseSize;
  
  // Get rotation in radians
  var rotationDeg = explicitDimensions.rotation || 0;
  var rotationRad = rotationDeg * Math.PI / 180;
  
  // Get opacity (0..1)
  var opacity = explicitDimensions.opacity !== undefined ? explicitDimensions.opacity : 1;
  
  // Save context and apply transformations
  this.ctx.save();
  
  // Apply transformations in correct order: translate → rotate → scale
  // Translate to center + offset
  this.ctx.translate(centerX + translateX, centerY + translateY);
  
  // Rotate around the centroid
  this.ctx.rotate(rotationRad);
  
  // Scale the artwork
  this.ctx.scale(scaleFactor, scaleFactor);
  
  // Apply global opacity
  this.ctx.globalAlpha = opacity;
  
  // Initialize style module
  if (typeof styleModule.init === 'function') {
    styleModule.init(this.ctx, cssWidth, cssHeight, renderingConfig || {});
  }
  
  // Generate data points (without explicit dimension offsets - they're now canvas transforms)
  // ... generate neutral data points ...
  
  // Render through style module
  styleModule.render(
    this.ctx,
    cssWidth,
    cssHeight,
    dataPoints,
    modifiedPalette,
    renderingConfig || {}
  );
  
  // Restore context
  this.ctx.restore();
};
```

**Key Changes:**
1. Calculate translate offset from X/Y dimensions (-1..1 normalized to canvas pixel offsets)
2. Calculate scale factor from Size dimension (100px = 1.0 scale)
3. Convert rotation to radians
4. Apply all transformations using `ctx.save()`, `ctx.translate()`, `ctx.rotate()`, `ctx.scale()`, `ctx.globalAlpha`
5. Render the artwork
6. Restore context with `ctx.restore()`

**Data Point Generation Changes:**
- The data points should be generated without applying explicit dimension offsets
- Since canvas transforms handle position/size/rotation/opacity, data points can be generated in a neutral coordinate space centered at (0.5, 0.5)
- This ensures the artwork is rendered centered, then transformed by the canvas operations
