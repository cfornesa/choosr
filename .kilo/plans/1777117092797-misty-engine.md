# Plan: Fix Canvas Centering and Rotation Origin

## Problem Statement

Two related issues in Manual Dimension mode:

1. **Default positioning is offset to lower right**: When studio.php loads, artwork appears in lower right instead of center
2. **Rotation happens around wrong point**: Rotation occurs around an "invisible center" rather than the data cluster center

## Root Cause Analysis

### Issue 1: Offset Positioning

**Current coordinate flow (BROKEN):**
```
1. Renderer: ctx.translate(cssWidth/2, cssHeight/2)  // Origin now at canvas center
2. Renderer generates data points: x=0.5, y=0.5  // "centered" around 0.5
3. Art style calculates: px = padX + normX * drawW
   - At normX=0.5: px = padX + 0.5 * drawW  // This is ~50% canvas width from origin!
4. Draw at (px, py) relative to translated origin
5. Result: Artwork at canvas_center + offset = LOWER RIGHT
```

**The Math:**
- Canvas center at (400, 300) for 800x600 canvas
- padX = 40 (5% of 800), drawW = 720
- At normX=0.5: px = 40 + 0.5*720 = 40 + 360 = 400
- Artwork drawn at origin(400,300) + (400, y) = (800, ...)
- That's off the right edge! (actually lower-right quadrant due to padding)

### Issue 2: Wrong Rotation Center

**Current behavior:**
- Rotation is applied via `ctx.rotate()` after `ctx.translate(cssWidth/2, cssHeight/2)`
- This rotates around the canvas center (now the origin after translate)
- But the data points are offset from the origin, so they rotate around a point that's not their center

**Expected behavior:**
- Rotation should happen around the center of the data cluster
- Or the data cluster should be centered at the rotation point

## Solution Options

### Option 1: Fix Data Point Generation (Recommended)

Change data points to be centered around 0 instead of 0.5:

**In renderer.js:**
```javascript
// OLD (broken):
var x = 0.5 + gridX * 0.6;  // Results in 0.2 to 0.8

// NEW (fixed):
var x = gridX * 0.6;  // Results in -0.3 to 0.3, centered at 0
```

**Pros:**
- Minimal changes to art styles
- Rotation naturally happens around data cluster center (at origin)
- Matches intuitive expectation: 0,0 = center

**Cons:**
- Art styles that expect 0-1 range need to handle negative values
- May need to adjust art style coordinate mapping

### Option 2: Fix Art Style Coordinate Interpretation

Keep data points at 0.5, but change how art styles interpret them:

**In art styles:**
```javascript
// OLD:
var px = padX + normX * drawW;

// NEW:
var px = (normX - 0.5) * drawW;  // Center at origin
```

**Pros:**
- Data points stay in 0-1 range
- No negative coordinate values

**Cons:**
- Must modify ALL 13 art styles
- Changes behavior for Data-Driven mode too (unless we branch)
- More complex logic in each style

### Option 3: Remove Canvas-Level Transforms, Apply Per-Point

Instead of canvas-level transforms, apply position/rotation/scale to each data point:

**Pros:**
- Full control over each point
- No transform matrix confusion

**Cons:**
- Massive refactor of renderer and all styles
- Loses the elegant canvas transform approach

## Correct Approach

The canvas transform chain is:
1. `ctx.translate(cssWidth/2, cssHeight/2)` — origin at canvas center
2. `ctx.rotate(rotationRad)` — rotate around origin
3. `ctx.scale(scaleFactor, scaleFactor)` — scale from origin

After these transforms, the coordinate system is:
- (0, 0) = canvas center
- (100, 0) = 100 pixels right of center
- (-100, 0) = 100 pixels left of center

So if data points have x in range [-0.5, 0.5], we want to map to [-spread, spread] pixels.

**In art style:**
```javascript
// OLD (0-1 range with padding):
var px = padX + normX * drawW;

// NEW (-0.5 to 0.5 range, no padding needed since canvas transform handles center):
var spreadX = drawW * 0.6;  // 60% of drawable width
var px = normX * spreadX;  // normX in [-0.5, 0.5], px in [-0.3*W, 0.3*W]
```

But this changes the coordinate contract for art styles. Let's verify Data-Driven mode compatibility.

## Data-Driven Mode Compatibility

In Data-Driven mode:
- Data points have x in [0, 1] from dataset normalization
- Art style maps: px = padX + normX * drawW

This works because there's NO canvas transform in Data-Driven mode (transforms are per-data-point based on values).

So we have two different coordinate systems:
- **Manual mode:** Canvas transform applied, data points should be in [-0.5, 0.5] range
- **Data-Driven mode:** No canvas transform, data points in [0, 1] range

The art styles currently expect [0, 1] range. We need to either:

A. Change Manual mode data points to [0, 1] but adjust the art style interpretation
B. Make art styles aware of which mode they're in
C. Fix the canvas transform approach differently

## Simplest Fix

The issue is that after `ctx.translate(cssWidth/2, cssHeight/2)`, the art style is still calculating positions as if the origin is at (0,0) top-left.

**Option A: Don't translate in renderer, let art styles handle centering**
Remove the canvas-level translate and handle everything in art styles.

**Option B: Pre-offset data points so they appear centered after translate**
If we translate origin to center, and want a point at canvas center, we need to draw at (0, 0) in the new coordinate system.

Currently data points are at x=0.5 which becomes px = padX + 0.5*drawW.

To make px = 0 (origin after translate), we'd need:
0 = padX + normX * drawW
normX = -padX / drawW ≈ -0.056

That's messy.

**Option C: Change art style to be origin-aware**
The art style should know the canvas has been transformed and adjust accordingly.

Actually, the simplest fix is to change the art style coordinate calculation to account for the fact that the origin is now at center:

```javascript
// OLD (origin at top-left):
var px = padX + normX * drawW;
var py = padY + normY * drawH;

// NEW (origin at center due to canvas transform):
// We want the artwork centered at origin
// Data points are at x=0.5, so we need to offset by -0.5 to center them
var px = (normX - 0.5) * drawW;
var py = (normY - 0.5) * drawH;
```

At normX=0.5: px = (0.5 - 0.5) * drawW = 0 ✓ (at origin/center)
At normX=0.2: px = (0.2 - 0.5) * drawW = -0.3 * drawW (left of center) ✓
At normX=0.8: px = (0.8 - 0.5) * drawW = 0.3 * drawW (right of center) ✓

This works! And it fixes both issues:
1. Centering: Data cluster centered at origin (canvas center after translate)
2. Rotation: Rotates around origin which is now the data cluster center
