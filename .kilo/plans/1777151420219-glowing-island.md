# Plan: Fix Manual Mode Art Style Rendering

## Problem Summary

After recent changes, 7 art styles fail to render in Manual Mode:
- neuralFlow, pixelMosaic, radialSymmetry, timeSeries, heatMap, scatterMatrix, barCode

Additionally, voronoiCells renders but is positioned incorrectly (lower right corner instead of center).

## Root Cause Analysis

### Issue 1: Single Point vs Multi-Point Rendering

The renderer's `renderUsingExplicitDimensions` method:
1. Generates **30 data points** for all styles (centered around x=0.5, y=0.5)
2. Applies canvas transforms BEFORE calling style render:
   - `ctx.translate(cssWidth/2 + translateX, cssHeight/2 + translateY)`
   - `ctx.rotate(rotationRad)`
   - `ctx.scale(scaleFactor, scaleFactor)`
3. Passes `renderConfig.manualMode = true`

**Working style (particleField)**:
- Iterates through ALL data points: `for (var i = 0; i < total; i++)`
- Applies positioning: `px = (normX - 0.5) * drawW`
- Draws many small particles

**Broken styles**:
- Check `if (isManualMode)` then only use `dataPoints[0]`
- Try to draw ONE large element instead of many small ones
- Missing visual output because single element is insufficient

### Issue 2: VoronoiCells Positioning

voronoiCells receives scattered points centered at 0.5, 0.5 but applies:
```javascript
px = (p.x - 0.5) * width;  // p.x is ~0.5, so px is ~0
py = (p.y - 0.5) * height; // p.y is ~0.5, so py is ~0
```

This should center correctly, but there's additional logic in `_drawVoronoi` that adds offsets:
```javascript
points.push({
  x: cx + Math.cos(angle) * r,  // cx is px from above
  y: cy + Math.sin(angle) * r   // cy is py from above
});
```

The issue is that `_drawVoronoiFromPoints` uses `ctx.getImageData(0, 0, w, h)` which reads from the transformed coordinate space, but the points have already been transformed by the renderer.

## Proposed Solution

### Option A: Match particleField Pattern (Recommended)

Update all 7 broken styles to:
1. Iterate through ALL dataPoints (not just `[0]`)
2. Draw multiple elements like particleField does
3. Use `(p.x - 0.5) * width` positioning which is correct

**Example fix for pixelMosaic**:
```javascript
if (isManualMode) {
  // Draw mosaic for each data point (like particleField)
  for (var i = 0; i < dataPoints.length; i++) {
    var p = dataPoints[i];
    var px = (p.x - 0.5) * width;  // Center around origin
    var py = (p.y - 0.5) * height;
    var tileSize = (p.size || 0.5) * MAX_SIZE * 0.1;
    // Draw smaller tiles since there are many points
    this._drawMosaic(ctx, px, py, tileSize, 5, opacity, rotation, color, colors);
  }
}
```

### Option B: Simplified Single-Element Mode

Keep using only `dataPoints[0]` but:
1. Draw at position (0, 0) since renderer already translated to center
2. Don't apply `(p.x - 0.5) * width` offset
3. Use `p.x` and `p.y` only for relative positioning within the element

**Example**:
```javascript
if (isManualMode) {
  var p = dataPoints[0];
  // Draw at origin - renderer already centered the context
  this._drawMosaic(ctx, 0, 0, tileSize, count, opacity, rotation, color, colors);
}
```

### VoronoiCells Specific Fix

The voronoi needs special handling because it uses `getImageData`:
1. Either don't apply canvas transforms for voronoi (renderer change)
2. Or account for transform in voronoi positioning
3. Or generate points in transformed coordinate space
