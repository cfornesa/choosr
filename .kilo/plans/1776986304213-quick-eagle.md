## 1. renderer.js Architecture

### Initialization & Sizing
- Constructor accepts `canvasElement` (DOM reference) and `options` object
- Canvas fills its container responsively using `ResizeObserver` (fallback to window resize event)
- Internal `devicePixelRatio` handling for crisp rendering on high-DPI displays
- Canvas CSS dimensions match container; internal pixel dimensions = CSS dimensions × DPR

### Input Acceptance
The `render()` method accepts five parameters:
1. `dataset` — normalized dataset object with `columns`, `row_count`
2. `columnMapping` — plain object mapping visual dimensions to column names
3. `paletteConfig` — `{ colors: [...], background: "#hex" (optional) }`
4. `renderingConfig` — style-specific defaults merged with user overrides
5. `styleKey` — string identifier ("particleField", "geometricGrid", "flowingCurves")

### Style Delegation
- Uses `artStyles.getStyle(styleKey)` to retrieve the appropriate style module
- Transforms raw dataset into `dataPoints` array (one object per row) with normalized visual dimension values
- Calls style module's `init()` once, then `render()` with the prepared data

### Resize Handling
- `resize()` method recalculates canvas dimensions based on container
- Clears and re-renders if data is cached (optional: store last render parameters)
- Debounced resize handler to prevent excessive redraws

### PNG Export
- `exportPNG(filename)` method:
  1. Calls `canvas.toBlob()` with MIME type `image/png`
  2. Creates temporary anchor element
  3. Sets `download` attribute to provided filename (or default "artwork.png")
  4. Triggers click and cleans up

### Clear Method
- `clear()` fills entire canvas with `paletteConfig.background` or default `#0d0d0d`

### Animation Control
- Rendering is synchronous for static outputs
- Uses `requestAnimationFrame` only when `renderingConfig.animate === true`
- No auto-play: `render()` must be explicitly called by user action

## 2. artStyles.js Role

### Registry Pattern
- Maintains internal registry object: `{ styleKey: styleModule }`
- Three built-in styles pre-registered on load

### Exported Functions
- `getStyle(styleKey)` — returns registered style module or throws if not found
- `listStyles()` — returns array of registered style keys
- `registerStyle(styleKey, styleModule)` — for future extensibility

### Style Module Interface
Each style module must implement:
```javascript
{
  // One-time setup called before first render
  init: function(ctx, width, height, renderingConfig) {},
  
  // Main rendering function
  render: function(ctx, width, height, dataPoints, paletteConfig, renderingConfig) {},
  
  // Optional: cleanup when switching styles
  cleanup: function() {}
}
```

### dataPoints Structure
Array of objects, one per data row:
```javascript
{
  x: 0.0-1.0,        // normalized or null if not mapped
  y: 0.0-1.0,
  size: 0.0-1.0,
  color: 0.0-1.0,    // maps to palette index
  opacity: 0.0-1.0,
  rotation: 0.0-1.0  // for applicable styles
}
```

## 3. Per-Style Rendering Approaches

### particleField.js
**Visual metaphor:** Field of glowing particles drifting through data space

**Dimension mappings:**
- `x` → horizontal position (0-1 mapped to canvas width with padding)
- `y` → vertical position (0-1 mapped to canvas height with padding)
- `size` → particle radius (minSize-maxSize from config)
- `color` → palette index selection (0-1 maps to palette.length indices)
- `opacity` → alpha value (0-1 mapped to minOpacity-maxOpacity)

**Fallbacks when dimensions unmapped:**
- `x` not mapped: deterministic grid layout based on row index
- `y` not mapped: deterministic vertical distribution based on row index
- `size` not mapped: default to config.defaultSize or mean of min/max
- `color` not mapped: cycle through palette sequentially
- `opacity` not mapped: default to config.defaultOpacity (0.8)

**Rendering approach:**
- Clear canvas to background
- For each data point: calculate pixel position, select color from palette, set globalAlpha, draw filled circle
- Optional: subtle glow effect using shadowBlur (canvas output exempt from C-02 constraints per C-03)

### geometricGrid.js
**Visual metaphor:** Tiled geometric cells with data-driven variation

**Dimension mappings:**
- `color` → palette index for cell fill
- `size` → scale factor for shape within cell (0 = empty, 1 = fills cell)
- `opacity` → cell fill alpha
- `rotation` → shape rotation in degrees (0-360 mapped from 0-1)

**Grid calculation:**
- `gridCols` and `gridRows` from renderingConfig
- Cell width = canvas.width / gridCols
- Cell height = canvas.height / gridRows
- Data rows map to grid cells sequentially (left-to-right, top-to-bottom)

**Fallbacks:**
- `color` not mapped: sequential palette cycling
- `size` not mapped: default 0.7 (partial fill looks intentional)
- `opacity` not mapped: default 1.0
- `rotation` not mapped: no rotation (0 degrees)

**Supported shapes:**
- Default: rectangle
- Configurable: circle, diamond, triangle
- Shape determined by renderingConfig.shape

**Rendering approach:**
- Calculate cell bounds for each data row
- Apply rotation transform at cell center
- Draw shape scaled by `size` factor within cell padding

### flowingCurves.js
**Visual metaphor:** Organic bezier curves flowing across canvas, guided by data

**Dimension mappings:**
- `color` → stroke color from palette
- `size` → stroke weight (lineWidth, minWeight-maxWeight from config)
- `opacity` → stroke alpha

**Curve generation:**
- `curveCount` from renderingConfig determines how many curves to draw
- Each curve uses multiple data rows for its control points
- Control points distributed horizontally across canvas at regular x-intervals
- Y positions derived from mapped column or deterministic distribution

**Curve structure:**
- Bezier curves with 4 control points each (start, two intermediates, end)
- Control points spaced horizontally across full canvas width
- Y values from data or smoothed interpolation

**Fallbacks:**
- `color` not mapped: each curve gets a palette color by index
- `size` not mapped: default stroke weight from config
- `opacity` not mapped: default 0.6 for layered transparency effect

**Rendering approach:**
- For each curve: beginPath, moveTo start point, bezierCurveTo with control points, stroke
- No fill — curves are gestural strokes
- Overlapping curves create layered depth

## 4. Data Normalization Strategy

### Numeric Column Normalization
```javascript
// Min-max normalization to 0-1 range
normalized = (value - columnMin) / (columnMax - columnMin)
```
- Handles edge case: if max === min, all values normalized to 0.5
- Null/undefined values: return null (style handles fallback)
- Non-numeric values in numeric column: attempt parseFloat, null on failure

### String/Categorical Column Mapping
- Each unique string value maps to a palette index
- Index = hash(value) % palette.length for deterministic assignment
- Alternative: sorted unique values assigned sequential indices
- Null/empty values: return null (style handles fallback)

### Boolean Column Mapping
- true → 1.0
- false → 0.0
- null/undefined → null

### Date Column Mapping
- Convert to timestamp
- Normalize within dataset's date range (min-max as with numbers)
- Returns 0-1 float

### Missing/Null Value Handling
- All normalizers return `null` for missing/null values
- Style modules check for null and apply dimension-specific fallbacks
- No data loss: raw value preserved in dataPoint for potential future use

### dataPoints Assembly (in renderer.js)
```javascript
// For each row index i:
dataPoint = {
  x: normalize(columns.find(c => c.column_name === columnMapping.x)?.values[i]),
  y: normalize(...),
  // etc for each visual dimension
}
```

## 5. Module Loading Strategy (No Build Tools)

Since the project uses no build tools, use browser-compatible patterns:

### Option A: Global Namespace Pattern (Recommended)
```javascript
// Each file wraps its code:
window.DataToArt = window.DataToArt || {};
window.DataToArt.Renderer = function(canvasElement, options) { ... };

// Usage in other files:
const renderer = new window.DataToArt.Renderer(canvas, options);
```

### Script Loading Order (HTML)
```html
<script src="src/canvas/styles/particleField.js"></script>
<script src="src/canvas/styles/geometricGrid.js"></script>
<script src="src/canvas/styles/flowingCurves.js"></script>
<script src="src/canvas/artStyles.js"></script>
<script src="src/canvas/renderer.js"></script>
<script src="src/app.js"></script>
```

```javascript
    registry['particleField'] = window.DataToArt.ParticleFieldStyle;
  }
  // ... etc
  
  return {
    getStyle: function(key) { ... },
    listStyles: function() { ... },
    registerStyle: function(key, module) { ... }
  };
})();
```

## 8. Assumptions Surfaced (Rule 1 Compliance)

1. **Assumption:** The container element for the canvas will always have a defined size when Renderer is initialized. If the container has no dimensions (e.g., display:none), canvas will have zero size.

2. **Assumption:** Data columns mapped to visual dimensions will have consistent data types within the column (no mixing of strings and numbers in a single column). The normalizer detects type per column but doesn't validate every cell.

3. **Assumption:** The browser supports ResizeObserver (modern browsers). A fallback to window.resize event is provided for older browsers.

4. **Assumption:** Color palette will always have at least one color. Empty palette will cause fallback to monochrome.

## 9. Design Options Considered (Rule 2 Gallery)

### Option 1: Immediate Rendering (Selected)
Renderer draws immediately when `render()` is called. No animation frame unless explicitly requested.
- **Pros:** Predictable, fast for small datasets, simpler mental model
- **Cons:** Large datasets may block UI thread briefly

### Option 2: Chunked Rendering with requestAnimationFrame
Always use rAF, render in chunks to avoid blocking
- **Pros:** Non-blocking even for large datasets
- **Cons:** Adds complexity, may feel sluggish for small datasets

### Option 3: Web Worker Offloading (Reframe)
Move data normalization to Web Worker, keep rendering on main thread
- **Pros:** Truly non-blocking data processing
- **Cons:** Requires Worker support, message passing overhead, overkill for Phase 1

**Decision:** Option 1 with explicit opt-in to animation via `renderingConfig.animate`. This aligns with DESIGN.md principle that rendering is "intentional and user-triggered."
