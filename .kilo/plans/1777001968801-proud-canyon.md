# Auth Panel Behavior Fix — Implementation Plan

## Problem Statement
The Data-to-Art Studio's Account panel currently shows Register/Login tabs and forms even when the user is logged in. The expected behavior is:
- **Logged OUT**: Show Register/Login tabs with appropriate forms, hide Logout button
- **Logged IN**: Hide all forms and tabs, show only "Logged in as USERNAME" + Logout button

---

## 1. Auth State Detection

**Approach:** Leverage the existing `data-username` attribute on `#dta-sidebar` that is already populated by PHP.

**Implementation:**
- In `src/app.js` `init()` function, read `document.getElementById('dta-sidebar').dataset.username`
- If username exists and is non-empty → `loggedIn: true`, `username: value`
- If no username or empty → `loggedIn: false`, `username: null`

**Why this approach:**
- Already implemented in studio.php: `data-username="<?php echo htmlspecialchars(get_current_username(), ENT_QUOTES, 'UTF-8'); ?>"`
- No additional API call needed on initial load
- Keeps auth state detection synchronous during init

---

## 2. UI Behavior Rules

### Logged OUT State (`loggedIn: false`)
- **Tabs**: Show Register/Login toggle buttons (`.dta-auth-toggle`)
- **Forms**: Show active form (default to Login), hide inactive form
- **Status text**: Show neutral label or empty (e.g., "Not logged in" or keep empty)
- **Logout button**: Hidden (`display: none`)
- **Summary text**: Show "Account"

### Logged IN State (`loggedIn: true`)
- **Tabs**: Hidden entirely (Register/Login buttons not relevant)
- **Forms**: Both Register and Login forms hidden
- **Status text**: Show "Logged in as USERNAME"
- **Logout button**: Visible (`display: block`)
- **Summary text**: Show "Logged in as USERNAME" (already works)

**Implementation mechanism:**
- Add a parent container class-based approach: toggle `.dta-auth-logged-in` on `#dta-auth-section`
- CSS rules will handle visibility based on this class
- JS `updateAuthUI()` function applies/removes the class and updates text content

---

## 3. Default Tab and Transitions

### Default Tab Decision
**When logged OUT:** Default to **Login** tab (not Register)
- Rationale: Returning users logging in is more common than new registrations
- Current behavior defaults to Register; this needs to change

**Implementation:**
- Call `_showLoginForm()` by default in `init()` when `loggedIn: false`
- Or add `data-default-tab="login"` to the container and read it

### Post-Login Transition
- After successful login response:
  1. Update `authState.loggedIn = true` and `authState.username = data.username`
  2. Call `updateAuthUI()` to immediately switch to logged-in view
  3. Forms hide, Logout button appears, tabs hide

### Post-Logout Transition
- After successful logout:
  1. Update `authState.loggedIn = false` and `authState.username = null`
  2. Call `updateAuthUI()` to switch to logged-out view
  3. Show Login tab by default
  4. Continue with existing redirect to `/index.php` after 1 second

## 4. Implementation Touchpoints

### A. studio.php (Markup Changes)
**Location:** The `<details id="dta-auth-section">` block

**Changes needed:**
1. Ensure `#dta-auth-section` exists as the container (already present)
2. Add a wrapper div around tabs and forms for easier visibility toggling:
   - New: `<div id="dta-auth-forms-container">` wrapping both `.dta-auth-toggle` and both forms
   - This allows hiding everything at once when logged in
3. Ensure `#dta-auth-status` exists for status messages (already present)
4. Ensure `#dta-logout-btn` exists (already present)

**No changes to:**
- Form IDs (keep `#dta-register-form`, `#dta-login-form` for JS submission handlers)
- Input field IDs (required for form submission)
- `data-username` attribute location

### B. css/app.css (New Rules)
**Add to end of file:**

```css
/* Auth panel state-based visibility */
#dta-auth-section.dta-auth-logged-in .dta-auth-toggle,
#dta-auth-section.dta-auth-logged-in .dta-auth-form {
  display: none !important;
}

#dta-auth-section:not(.dta-auth-logged-in) #dta-logout-btn {
  display: none;
}

/* When logged in, adjust spacing */
#dta-auth-section.dta-auth-logged-in #dta-auth-status {
  margin-bottom: 1rem;
}
```

**Preserve existing:**
- All color schemes, shadows, fonts
- `.active` class styling for tabs
- Form input styling

### C. src/app.js (Logic Changes)

**New state object (inside IIFE):**
```javascript
var _authState = {
  loggedIn: false,
  username: null
};
```

**New function: `_updateAuthUI()`**
- Checks `_authState.loggedIn`
- Toggles `.dta-auth-logged-in` class on `#dta-auth-section`
- Updates `#dta-auth-status` text
- Updates `#dta-auth-summary` text
- Shows/hides logout button

**Modified `init()`:**
- After DOM references are cached, detect auth state from `data-username`
- Populate `_authState`
- Call `_updateAuthUI()`
- If not logged in, call `_showLoginForm()` (sets default tab to Login)

**Modified `_onLoginSubmit()` success handler:**
- After `data.success`, update `_authState.loggedIn = true`
- Call `_updateAuthUI()` instead of manual DOM updates
- Keep existing: `_loginForm.reset()`, `loadDatasetList()`

**Modified `_onLogoutClick()` success handler:**
- After `data.success`, update `_authState.loggedIn = false`
- Call `_updateAuthUI()`
- Keep existing: redirect to `/index.php`

## Assumptions Surfaced

1. **Assumption:** The `data-username` attribute on `#dta-sidebar` reliably indicates logged-in state because `studio.php` redirects unauthenticated users to `index.php`. If this protection changes, the auth detection would need an API fallback.

2. **Assumption:** Defaulting to Login tab (vs Register) for logged-out users improves UX. If analytics show more registrations than logins, this should be reconsidered.

3. **Assumption:** Hiding the tabs entirely when logged in (rather than disabling them) is the cleanest UX. An alternative would be to show disabled tabs with a "Already logged in" message, but this adds visual noise.
