# Plan: Fix Manual Dimension Visualization Issues

## Problem Summary

Two distinct issues were identified in the Manual Dimension mode for art styles:

### Issue 1: Blank Canvas After Selecting Certain Styles
**Affected styles:** `neuralFlow`, `voronoiCells`, `timeSeries`

**Root Cause:** These style files reference an undefined variable `renderingConfig` instead of the function parameter `rc`. This causes:
- The `manualMode` check to always fail
- Position calculations to use incorrect logic
- In some cases, the styles don't render at all due to JavaScript errors

**Evidence from code review:**
```javascript
// neuralFlow.js line 38 - BUG: 'renderingConfig' is undefined, should be 'rc'
if (renderingConfig && renderingConfig.manualMode) {
    px = (p.x - 0.5) * width;
    py = (p.y - 0.5) * height;
}
```

Same issue exists in:
- `voronoiCells.js` line 30
- `timeSeries.js` line 38

### Issue 2: Artwork Centered at (-1, -1) Position
**Affected styles:** `radialSymmetry`, `pixelMosaic`, `heatMap`, `scatterMatrix`, `barCode`

**Root Cause:** Same variable naming bug - these files reference `renderingConfig` instead of `rc`. When the manual mode check fails, the styles fall back to data-driven positioning logic:
```javascript
// When manualMode check fails, falls back to:
symX = cx + p.x * width/2;  // = width/2 + 0.5 * width/2 = 0.75 * width
symY = cy + p.y * height/2; // = height/2 + 0.5 * height/2 = 0.75 * height
```

This positions the artwork at 3/4 of canvas dimensions instead of center.

**Evidence from code review:**
```javascript
// radialSymmetry.js line 32 - BUG: 'renderingConfig' should be 'rc'
if (renderingConfig && renderingConfig.manualMode) {
    symX = (p.x - 0.5) * width;
    symY = (p.y - 0.5) * height;
}
```

Same issue exists in:
- `pixelMosaic.js` line 30
- `heatMap.js` line 32
- `scatterMatrix.js` line 30
- `barCode.js` line 33

## Comparison with Working Style

**particleField.js** correctly uses the parameter name:
```javascript
render: function(ctx, width, height, dataPoints, paletteConfig, renderingConfig) {
    // ... correct usage throughout ...
    if (renderingConfig && renderingConfig.manualMode) {
        px = (normX - 0.5) * drawW;
        py = (normY - 0.5) * drawH;
    }

```

## Proposed Solution

### Option A: Rename parameter to `renderingConfig` (Recommended)
Change the function parameter name from `rc` to `renderingConfig` in all 8 affected files. This:
- Makes the code self-documenting (no abbreviation)
- Matches the working style's naming convention
- Is a simple find-and-replace operation

### Option B: Replace `renderingConfig` with `rc`
Replace all occurrences of `renderingConfig` with `rc` in the 8 affected files. This:
- Uses shorter parameter name consistently
- Requires more changes (multiple occurrences per file)

## Implementation Steps

1. **Fix parameter naming** in all 8 affected style files:
   - Change `function(ctx, width, height, dataPoints, palette, rc)` 
   - To: `function(ctx, width, height, dataPoints, palette, renderingConfig)`

2. **Verify the fix** by checking that:
   - All references to `rc` within each function are replaced with `renderingConfig`
   - No stray `renderingConfig` references remain (would cause errors)

3. **Test scenarios**:
   - Manual mode with neuralFlow, voronoiCells, timeSeries should now render
   - Manual mode with radialSymmetry, pixelMosaic, heatMap, scatterMatrix, barCode should now be centered
   - Data-driven mode should continue to work for all styles
